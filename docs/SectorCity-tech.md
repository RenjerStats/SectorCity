# «TerraForm Drive» — техническая архитектура и стек

> Дополнение к ТЗ. Здесь — конечный набор библиотек, развилки выбора, типичные накладки и способы их решения, а главное — **как синхронизировать 3D-сцену и DOM-интерфейс** (обводка по наведению, окно над зданием по клику, как в игре).

---

## 1. Главная архитектурная идея: store как мост

В проекте сосуществуют два принципиально разных мира:

- **Императивный 3D-мир** (Three.js / WebGL): цикл рендера, `InstancedMesh`, камера, raycaster. Состояние меняется покадрово, напрямую дёргать его из реактивного UI нельзя.
- **Декларативный DOM-мир** (тулбар, хлебные крошки, тултипы, контекстное меню, легенды): реагирует на изменение состояния.

Соблазн — связать их напрямую (raycaster дёргает DOM, кнопка дёргает сцену). Это путь к спагетти и гонкам. **Правильно: оба мира общаются только через единый стор (single source of truth). Никто не трогает другого напрямую.**

```
┌───────────────────────────────────────────────┐
│            UI Layer (DOM)                       │  подписан на стор
│  toolbar · breadcrumbs · tooltip · context menu │
│  legends · progress              (Svelte + TS)  │
└──────────────┬──────────────────────────────────┘
               │ читает state / шлёт intents
┌──────────────▼──────────────────────────────────┐
│      App State — единый источник правды           │
│  store (nanostores/zustand)  +  state machine     │
│  hoveredId · selectedId · currentPath · mode      │
│  data · scanStatus · cameraTarget                 │
└──────────────┬──────────────────────────────────┘
               │ читает state / пишет hovered, selected
┌──────────────▼──────────────────────────────────┐
│       3D Render Layer (императивный)              │
│  Three.js · InstancedMesh · MapControls           │
│  raycaster · highlight-mesh · tween · projection  │
└──────────────┬──────────────────────────────────┘
               │ Tauri IPC (commands + event stream)
┌──────────────▼──────────────────────────────────┐
│              Rust Backend                          │
│  Scanner(jwalk) → Aggregator → Snapshot(SQLite)   │
│  → Classifier ;  commands + progress events        │
└────────────────────────────────────────────────────┘
```

**Поток наведения (hover → обводка):**
1. `pointermove` (троттлинг!) → `raycaster.intersectObject(instancedMesh)` → `instanceId`.
2. `instanceId` → `nodeId` (держим массив `index → node`).
3. Если изменилось — `store.hoveredId = nodeId`.
4. **3D-слой** в своём кадровом цикле читает `hoveredId`, двигает highlight-mesh на матрицу инстанса и/или ставит emissive-подсветку.
5. **UI-слой** реагирует на `hoveredId`, показывает тултип; его позиция обновляется покадрово через проекцию (см. §4).

**Поток клика (click → окно над зданием):**
1. `pointerup` без drag → raycast → `nodeId`.
2. Диспатч в state machine: файл → `SELECT`; район (папка) → `DRILL`.
3. `SELECT`: `store.selectedId = nodeId`, режим → `selected`; UI рисует окно-карточку, заякоренное над зданием (проекция мировой точки в экранные пиксели).
4. `DRILL`: режим → `zooming`; tween камеры к центру района (~500 мс); по завершении — IPC-запрос дочернего уровня → пересборка города → режим → `idle` на новом уровне → обновление хлебных крошек.

Ключевой принцип: **высокочастотное (позиция оверлея покадрово) делаем императивно через ref/прямую запись в DOM; низкочастотное (что выбрано, какой режим) — через реактивный стор.** Не гонять реконсиляцию фреймворка каждый кадр — это смерть производительности.

## 3. Зафиксированные решения по состоянию и событиям

Все развилки управления состоянием/событиями закрыты. Ниже — что выбрано и почему отвергнуты альтернативы.

### A — UI-фреймворк: **Svelte** ✅

Компилируется в ноль, чистая реактивность, сторы идеально дружат с императивным циклом рендера. DOM здесь — обвязка (тулбар, крошки, тултип, меню, легенды), звезда — 3D, поэтому тяжёлый фреймворк не нужен.
*Отвергнуто:* vanilla TS (больше ручного DOM); React + R3F — **R3F дерётся с инстансингом и внешней d3-раскладкой** (тысячи компонентов и реконсиляция на каждом drill), поэтому город строим императивным Three.js, а не декларативным R3F.

### B — стор-мост: **nanostores** ✅

Крошечный, framework-agnostic. К одному atom подписаны и императивный Three.js-цикл (`atom.subscribe(...)`), и Svelte-UI. **Стор — единственная точка контакта 3D и UI.**
*Отвергнуто:* zustand (тоже годится, но nanostores легче и нейтральнее к фреймворку).

### C — машина режимов: **discriminated union + явные переходы** ✅

Без новой зависимости, типобезопасно, закрывает гонки (drill во время незавершённого зума, завершение скана посреди навигации).

```ts
type AppMode =
  | { kind: 'scanning'; progress: number }
  | { kind: 'idle'; path: string }
  | { kind: 'hovering'; path: string; hoveredId: NodeId }
  | { kind: 'selected'; path: string; selectedId: NodeId }
  | { kind: 'zooming'; from: string; to: string };
```

Невалидные действия (например, `DRILL` в состоянии `zooming`) отсекаются гардом на уровне типов.
*Отвергнуто:* россыпь булевых флагов (гонки); XState (избыточен для срока).

### D — события ввода: **нативные `pointer` + ручной троттлинг** ✅

`requestAnimationFrame`-гейт на raycast — достаточно, без зависимостей.
*Отвергнуто:* RxJS (мощно, но тяжело и избыточно для нашего набора событий).

## 4. Окно над зданием «как в игре» — как сделать правильно

Окно/тултип **не рисуем в WebGL** (текст в WebGL мылит и недоступен скринридерам). Рисуем как обычный DOM-оверлей, заякоренный к 3D-точке.

Механика:
1. Берём мировую позицию верхушки здания (центр + высота).
2. Проецируем в clip-space: `vector.project(camera)`.
3. Переводим в пиксели: `x = (v.x*0.5+0.5)*width`, `y = (-v.y*0.5+0.5)*height`.
4. Ставим абсолютно спозиционированный DOM-элемент в эти координаты (через `transform: translate(...)`).
5. **Контент** окна (путь, размер, даты) приходит из стора; **позиция** обновляется покадрово императивно через `ref`, пока движется камера.

**Реализация проекции: вручную (как выше)** ✅ — **зафиксировано (G)**: максимальный контроль, чистая интеграция со Svelte-стором.
*Отвергнуто:* `CSS2DRenderer` (автоматизирует, но меньше контроля); `@react-three/drei <Html>` (только для R3F, который мы не берём).

Накладка: DOM-оверлеи **не уважают 3D-глубину** (карточка здания за «горой» всё равно поверх). При навигации «один уровень за раз» это почти не мешает; если понадобится — гасить оверлей, когда здание перекрыто (проверка глубины raycast’ом).

## 5. Типичные накладки и как их решать

### 5.1. Обводку одного инстанса `InstancedMesh` нельзя сделать через OutlinePass
`OutlinePass`/`postprocessing` обводят **объект целиком**, а не отдельный инстанс. Город — это один `InstancedMesh`, поэтому обвести только наведённое здание им не выйдет.
**Решение (дёшево):** держать **один отдельный highlight-mesh** (рёберная геометрия `EdgesGeometry` или чуть увеличенный бокс с backface-обводкой) и на каждом изменении `hoveredId` ставить его трансформ = матрице наведённого инстанса. Плюс опционально `instancedMesh.setColorAt(id, ...)` для emissive-тинта. Никакого пост-процессинга, один лишний меш.

### 5.2. Raycast по `InstancedMesh` тормозит при тысячах инстансов
Луч тестирует все инстансы на каждом `mousemove`.
**Решения по нарастанию:** (1) троттлить raycast до ~30–60 раз/с через rAF-гейт; (2) при росте — `three-mesh-bvh` для ускорения; (3) в пределе — GPU-picking (рендер ID в 1×1 буфер под курсором). Для «топ-N зданий на уровень» хватает (1).

### 5.3. Покадровое обновление стора = смерть реконсиляции
Если писать позицию оверлея в реактивный стор каждый кадр — фреймворк перерисовывается 60 раз/с.
**Решение:** разделить **идентичность/контент** (низкочастотно, в стор) и **позицию** (высокочастотно, императивно через `ref` в обход реконсиляции).

### 5.4. Поток событий прогресса скана топит UI
Скан шлёт тысячи событий.
**Решение:** батчить/троттлить прогресс на бэкенде или фронте (обновлять не чаще, чем раз в ~100 мс).

### 5.5. Утечка GPU-памяти при drill
Пересборка города без освобождения старой геометрии/материалов течёт.
**Решение:** на смене уровня явно `geometry.dispose()` / `material.dispose()` / `texture.dispose()`; переиспользовать буферы инстансов, где можно.

### 5.6. Несовпадение систем координат d3 ↔ Three.js
d3-treemap даёт `x0,y0,x1,y1` в 2D (y вниз), Three.js — 3D (y вверх).
**Решение:** зафиксировать конвенцию маппинга один раз: treemap-x → world-x, treemap-y → world-z, размер → footprint, устаревание → world-y (высота). Вынести в один модуль `layoutToWorld()`.

### 5.7. Раздувание IPC-payload
**Решение (уже в дизайне):** один уровень + превью +1, tail-агрегация на бэкенде. Запас — бинарный канал (`rmp-serde`/`bincode`) вместо JSON.

### 5.8. Различия WebView между платформами
Tauri использует системный WebView (WebView2 / WKWebView / WebKitGTK); поведение WebGL может чуть отличаться.
**Решение:** Windows-first снимает это на старте; кросс-платформенную проверку WebGL вынести в отдельную фазу.

### 5.9. `atime` недостоверен
**Решение (из ТЗ):** база высоты — `mtime`; `atime` — только как уточнение при достоверности.

## 7. Итоговый стек (фиксация)

**Бэкенд:** Rust + Tauri v2; `jwalk` (скан) · `rayon` (агрегация) · `rusqlite` (снимок/diff) · `serde`/`serde_json` (+`rmp-serde` про запас) · `tokio`/`tokio-util` (отмена) · `thiserror`/`anyhow` · `tracing`; плагины `dialog`, `opener`; `trash`, `blake3` (фаза 3).

**Фронтенд:** TypeScript + Vite + **Svelte**; `three` (рендер, `InstancedMesh`) · `d3-hierarchy` (squarified) · `@tauri-apps/api` (IPC) · `@tweenjs/tween.js` (камера) · `MapControls` · `nanostores` (стор-мост) · discriminated-union state machine (без зависимости) · опц. `troika-three-text`, `three-mesh-bvh`.

**Несущий принцип:** императивный 3D и декларативный DOM общаются **только через стор**; высокочастотное — императивно, низкочастотное — реактивно.
