# ТЗ практики: «SectorCity»
### Desktop-приложение для визуализации размера папок на диске

> **Цель проекта:** инструмент анализа дискового пространства, где приоритет — **наглядность, читаемость информации и удобство использования**, а не декоративная красота. Визуализация должна честно отвечать на вопрос «что занимает место» быстрее и понятнее, чем плоские таблицы и стандартные treemap-утилиты.

---

## 1. Зафиксированные решения

| Аспект | Решение | Почему |
|---|---|---|
| **Метафора** | 2.5D «город в городе»: здание = файл, район = папка (рекурсивно) | 3-е измерение несёт информацию, а не декор; рекурсия отражает структуру ФС |
| **Кодирование размера** | Честный масштаб (площадь основания = размеру) + tail-агрегация + drill-down | Площадь — сильнейший канал для сравнения величин; искажение масштаба = ложь о данных |
| **Кодирование высоты** | Высота = устаревание (база `mtime`) | Большое + высокое + категория-мусор = мишень для очистки, считывается одним взглядом |
| **Кодирование цвета** | Цвет = категория содержимого (не размер) | Категория помогает узнаванию; палитра colorblind-safe |
| **Навигация** | Один уровень за раз + плавный зум с превью на +1 уровень | Убирает окклюзию, держит сцену управляемой на больших дисках |
| **ИИ** | Нет. Только правила/эвристики | Снимает риски приватности путей, латентности и стоимости; правила дают ~90% пользы |
| **Стек** | Rust-сканер + Tauri + Three.js + TypeScript | Быстрый нативный скан + бесплатный squarified-treemap (`d3-hierarchy`) + дешёвый HTML/CSS UI |

## 2. Ключевые принципы читаемости

Заложены в дизайн как требования, не как пожелания:

- **Один визуальный канал на одну переменную.** Площадь → размер. Высота → устаревание. Цвет → категория. Каналы не перегружаются и не дублируются.
- **Правда всегда доступна.** По наведению/выбору — точный размер, полный путь, число файлов, даты. Реальные имена видны всегда.
- **Честный масштаб без искажений.** Проблема мелких файлов решается агрегацией, а не раздуванием.
- **Стабильная детерминированная раскладка.** Повторный скан той же ФС даёт ту же геометрию → работает пространственная память пользователя.
- **Быстрая обратная связь.** Прогрессивный рендер по мере скана, без ожидания полного обхода диска.
- **Управляемая камера.** Орбита + зум с ограниченным наклоном, кнопка «сбросить вид». Никакого дезориентирующего свободного полёта.

## 3. Метафора «город в городе»

- **Здание = файл.** Площадь основания пропорциональна размеру, высота — устареванию.
- **Район = папка.** Отделяется тинтом земли, бордюром/зазором и категориальным цветом. Внутри — свои здания и под-районы (рекурсия).
- **Плотность застройки** внутри района — высокая (squarified тайлит без зазоров), **но между районами — небольшие отступы** (padded treemap). Это осознанный размен плотности на читаемость границ.
- **Клик по зданию** → контекстное меню (детали, действия).
- **Клик по району** → плавный зум на уровень ниже; этот район становится новым «городом».

### Плавный зум с превью (+1 уровень)

«Очертания района до открытия» и «открытый вид» — это **одна и та же раскладка на разных уровнях детализации**:

1. Бэкенд отдаёт текущий уровень **+ детей видимых районов** (глубина +1), всё с tail-агрегацией → payload ограничен.
2. Раскладка — **вложенный treemap**: внутри прямоугольника района рекурсивно разложены его дети. Это и есть «очертания», посчитанные заранее.
3. Рендер через **LOD**: издалека район = тинтованный силуэт из грубых блоков; при приближении те же блоки уточняются в отдельные здания. Геометрия не пересобирается → нет рывка, силуэт плавно «проявляется».
4. Зум — анимация камеры (ease position + target, ~400–700 мс) к центру района; по прилёте уровень коммитится как новый корень, обновляются хлебные крошки.
5. Живыми держим только уровни «текущий ± 1»; глубже — агрегированные массы. Превью считаем лениво и только для достаточно крупных районов.

### Проблема мелких файлов (без искажения масштаба)

Всё ниже порога схлопывается в один блок **«Прочее (N файлов)»**, чья площадь = честной сумме содержимого. Разворачивается по клику, доступен через drill-down. Честность сохранена, сцена не зашумлена.

## 4. Стек и обоснование

- **Сканер: Rust** (`jwalk` для параллельного обхода) — нативная скорость на миллионах файлов, не блокирует UI.
- **Оболочка: Tauri** (Rust-бэкенд + webview-фронт) — легче Electron, нативный доступ к ФС.
- **Рендер: Three.js** поверх WebGL (GPU). Город из тысяч коробок рисуется через **`InstancedMesh`** — производительность нативная; узким местом был бы только наивный JS, не GPU.
- **Раскладка: `d3-hierarchy`** — squarified-treemap из коробки (экономит дни разработки).
- **UI: TypeScript** — контекстные меню, тултипы, легенды, поиск на HTML/CSS дёшевы и сопровождаемы.

**Почему не pure-Rust (Bevy):** концептуально чище (нет границы сериализации), но treemap пришлось бы писать самому, а «скучный» UI на Rust заметно дороже HTML/CSS. Для ограниченной по срокам практики выбран путь с меньшим риском и быстрым выходом на MVP. Переход на Bevy оправдан только при уверенной Rust-команде, осознанно выбирающей single-language ради 3D.

## 5. Архитектура

**Поток данных:** `Scanner → Aggregator → Snapshot → IPC → Layout → Renderer`, плюс сбоку `Classifier` и `Interaction`.

Единственная граница сериализации (Rust ↔ webview) нейтрализована тем, что наружу уходит только текущий уровень + превью, уже агрегированные по хвосту.

### Слои

1. **Scanner (Rust):** асинхронный, стримящий, отменяемый, с прогрессом.
2. **Aggregator (Rust):** свёртка размеров и счётчиков снизу вверх.
3. **Snapshot (Rust):** снимок ФС в SQLite — быстрое переоткрытие + основа для diff.
4. **IPC (Tauri):** ленивая отдача по уровням.
5. **Layout (фронт, squarified):** прямоугольники текущего уровня + tail-агрегация.
6. **Classifier (Rust):** категории и флаги кандидатов на удаление по правилам.
7. **Renderer (Three.js):** экструзия, цвет, высота, лейблы, LOD.
8. **Interaction:** hover, drill, поиск-подсветка, фильтры, контекстные меню.

### Контракт данных (узел сканера)

```
ScanNode {
  path, name, is_dir,
  size: u64,         // для папок — свёрнутая сумма потомков
  mtime, atime: i64, // для высоты-устаревания
  child_count: u32,
  category: enum,    // присваивает Classifier
  flags: [...]       // напр. CleanupCandidate
}
```

### IPC-команды (Tauri)

- `start_scan(root)` — стримит прогресс, пишет снимок;
- `get_level(path, top_n, depth=2)` — дети папки + дети районов (превью +1), с tail-агрегацией в «Прочее»;
- `get_node_detail(path)` — полная правда для тултипа/меню;
- `search(query)` — пути для подсветки.

### Инженерная оговорка по устареванию

`atime` на Windows часто отключён (`NtfsDisableLastAccessUpdate`), на Linux `relatime` делает его неточным. **База высоты — `mtime`**; `atime` используется как уточнение только при достоверности. Это честнее, чем притворяться, что «давность доступа» всегда доступна.

## 6. Риски и митигации

| Риск | Митигация |
|---|---|
| `atime` ненадёжен | База на `mtime`, `atime` — только как уточнение |
| Залоченные/системные файлы | Пропускать, считать и показывать число ошибок |
| Циклы по junction/symlink | Не следовать reparse points |
| Долгий скан больших дисков | Стрим + прогресс + отмена |
| Раздувание IPC-payload | Tail-агрегация + один уровень; запас — бинарный `Channel` (bincode/MessagePack) вместо JSON |
| Расползание скоупа | Жёсткие гейты по фазам, MVP как страховочный пол |
